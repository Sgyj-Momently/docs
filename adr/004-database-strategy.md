# ADR 004 - Database Strategy

## 상태

승인됨

## 날짜

2026-04-20

## 맥락

오케스트레이터는 워크플로 상태, 단계별 실행 결과, 실패 정보, 재시도 이력을 안정적으로 저장해야 한다.
또한 높은 트래픽과 향후 분산 배포를 가정하면, 단순 메모리 저장소나 파일 기반 저장만으로는 운영 요구사항을 만족하기 어렵다.

후보 방향은 아래와 같았다.

- 메모리 저장소 유지
- 관계형 DB 중심 설계
- NoSQL 단독 설계

## 결정

- 기본 운영 DB는 `PostgreSQL`로 한다.
- 워크플로 메타데이터와 단계 상태 이력은 PostgreSQL에 저장한다.
- 큰 JSON 산출물과 문서 결과물은 DB 본문에 직접 누적하지 않고 artifact 저장소로 분리한다.
- 캐시, 분산 락, 단기 상태 공유가 필요할 때만 `Redis`를 보조 계층으로 추가한다.

## 이유

- 워크플로 상태 전이와 재시도 정책은 트랜잭션 일관성이 중요하다.
- PostgreSQL은 관계형 모델, 인덱스, JSONB, 운영 생태계 측면에서 균형이 좋다.
- 스프링과의 궁합이 좋고, 높은 트래픽 환경에서도 널리 검증된 선택지다.
- 대용량 산출물을 DB에 직접 넣기 시작하면 비용과 성능, 백업 복잡도가 빠르게 커진다.

## 결과

- 현재의 `InMemoryWorkflowRepository`는 초기 개발용 임시 구현으로 본다.
- 추후 `JPA` 또는 적절한 persistence adapter로 PostgreSQL 연동을 추가한다.
- 오케스트레이터 설계 문서에 메타데이터 저장과 artifact 분리 원칙을 명시한다.
